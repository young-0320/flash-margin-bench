## 작업 내용

- **원격 실행 시도 (세은 노트북, Windows + Vivado 2025.2)**: xsdb로 프로그래밍은 성공(DONE 점등)했으나 캡처가 BEGIN을 못 들음. 유력 원인 = FT2232 듀얼 채널(A=JTAG, B=UART)이 Windows에서 구분 없이 COM으로 잡혀 JTAG 채널을 청취한 것(미확정 — 다른 COM 미시도 철수). 리눅스는 UART만 ttyUSB1로 바인딩되어 모호성 없음. **점퍼 접촉 불량이 중첩**되어 있었음(귀가 후 첫 실행이 timeout으로 입증) — 문제 2개 중첩 시 하나를 고쳐도 증상이 남아 "재현성 문제"로 오인됨. 측정 시스템 자체는 양쪽에서 동일 동작.
- **line_dead_unknown 오발 발견**: 940스텝 무에러 후 벽 진입 중 스텝 952=953의 100읽기 에러 벡터가 비트 단위 동일 → 진단 발동, 스윕 중단. 자가진단의 가정("창 밖 = 노이즈라 연속 스텝 동일 불가")이 **저노이즈 결정론 채널(루프백)의 벽 평탄부에서 깨짐** — 엣지가 다음 Δφ 안에 없으면 틀리는 비트 집합이 불변. 수정 = `err_reads==N && !low_seen` 게이트: 죽은 라인은 저에러 스텝을 낼 수 없으므로(설계 의도 "스텝 2에서 중단"도 low_seen=0 구간) 원 목적 보존, 벽 도달 정상 스윕만 오발 면제.
- **θ 확정 체크리스트**: ② BER floor(상한 3.7e-9) PASS / ③ σⱼ 29.1·29.2ps → MC 재실행 N=100 재확인 PASS / ④ 과분산 F=0.48 PASS / ⑤ 프레이밍 슬립 없음 PASS / **① probit 직선성 FAIL** (좌 0.993 / 우 0.9742, 기준 0.98)

## Windows에서 재현하는 법 (원격/타 머신 실행)

코드 수정 불요 — PS C는 Zynq 자신의 UART에 쓸 뿐이고 tcl은 JTAG 전용이라 PC 포트 개념이 없다. 포트가 등장하는 유일한 곳 = 캡처 스크립트의 `--port` 인자 (개념: `docs/concepts/9`).

1. 풀 Vitis 불요 — 풀 Vivado가 있으면 **xsdb** 동봉 (`<설치>\Vivado\<버전>\bin\xsdb.bat`), 없으면 Vivado Lab Edition(~2GB). 타 버전(2023.1/2025.2) xsdb로 2024.2 산출물 프로그래밍 동작 확인
2. 파일 5개를 구조 유지 복사(bit·elf·ps7_init.tcl·program_g0.tcl·캡처) 또는 리포 클론 + `build/` 산출물 복사
3. UART COM 판별: 장치 관리자에서 부모가 "USB Serial Converter **B**"인 COM. 모호하면 후보 전부 실증 — 캡처 후 재프로그래밍 → **BEGIN이 수 초 내** 뜨는 쪽 (오판 무해)
4. `python host/capture/sweep_uart_capture.py --target loopback --port COM<N>` (클론 밖 실행이면 `--outdir .` 추가) → xsdb로 program_g0.tcl → 이하 동일

## 미결 의사결정

- **θ=10⁻² 확정 보류** — 체크리스트 ① 우벽 probit FAIL. 원인 가설: line_dead 오발과 동일 뿌리인 **벽의 결정론적 평탄부**(BER~0.1대 선반)가 가우시안 erfc 모델을 위반. 선택지 = (a) 피팅 범위 규칙 조정(BER 상한 제한; 사전 등록 규칙 변경이라 MC 주석·보고서 문구 동시 갱신 필요) / (b) 보류 상태로 실칩 진행(θ는 잠정 유지) / (c) 재측정. **3인 논의 필요.**
- line_dead 무효 사유의 §6 정식화 논의 시 `!low_seen` 의미 변화 공유 (지민 — 검증 분담 ⑥ 관련)
- C8 게재 승격(plots → docs/reports/)은 θ 확정 후로 보류
