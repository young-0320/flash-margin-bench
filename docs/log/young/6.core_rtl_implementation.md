# 6. core 블록 RTL 구현 (MMCM 위상 제어 + AXI-Lite 레지스터 파일)

- 작성일: 2026-07-07

## 산출물

- `fpga/rtl/core/core_top.v` — core 블록 최상위. 하위 결선 + contract §2 경계 포트. 통합 전제: AXI 인터커넥트/Zynq GP ACLK에 clk_core 출력을 물림(제어 전 도메인 clk_core, CDC 0)
- `fpga/rtl/core/core_axil_regs.v` — AXI-Lite 슬레이브 + 레지스터 맵 v0(contract §3, ID=0x4D42_0100). 규약 집행 지점: R1·R2·R4 명령 게이팅, R5 DONE sticky, R10·R11 sticky·자동 클리어
- `fpga/rtl/core/core_phase_ctrl.v` — PSEN/PSINCDEC/PSDONE 핸드셰이크 FSM(UG472: PSEN 1사이클, PSDONE까지 재명령 금지) + PS_BUSY + PHASE_POS(signed 32) 누적
- `fpga/rtl/core/core_mmcm.v` — MMCME2_ADV 래퍼(M=9·D=1 → VCO 1,125MHz, CLKOUT0/1 ÷45 = 25MHz, CLKOUT1만 USE_FINE_PS, 피드백 내부 직결). 프리미티브를 이 파일에만 격리 — 시뮬은 스텁 대체
- `sim/smoke/tb_core_smoke.v` + `unisim_stub.v` — iverilog 스모크 TB (실행법은 파일 헤더). cocotb 회귀 확립 전까지 core 수정 시 최소 회귀

## 작업 내용

- 모듈 분할 원칙: 프리미티브(core_mmcm) / 순수 로직(나머지) 분리 — iverilog+cocotb 사다리(박지민)에서 MMCM 스텁만 갈아끼우면 core 전체가 순수 Verilog로 시뮬 가능
- 계약 미정의 지점 2건을 구현에서 확정 (아래 미결 참조):
  - **CTRL 다중 비트 쓰기 = 실행 거부 + `STATUS.CMD_ERR`(bit6, sticky, 리셋으로만 클리어)** — 호스트 시퀀스는 구조적으로 한 번에 한 명령이므로 조합은 항상 호스트 버그. 처음엔 조용한 무시로 구현했으나 R11과 같은 "시끄러운 거부" 철학으로 변경(7/7 리뷰 반영). 순차 실행안은 요청 없는 기능(명령 큐 FSM)이라 기각. 리셋 클리어인 이유: 측정 상태가 아니라 소프트웨어 버그 검출기 — 스윕 종료 시 1회 확인해 무효 판정에 쓰는 증거물이므로 다음 START가 지우면 안 됨. R4의 busy 중 무시는 계약 문언 그대로 조용히 유지
  - **sticky 클리어 시점 분리**: DONE은 START 수락 시(R5 문언), TIMEOUT·CFG_ERR은 meas_busy 상승(= "유효 START", R11 문언) 시 클리어 — 거부된 START가 이전 TIMEOUT 증거를 지우지 못함
- 검증 3층:
  1. **iverilog 스모크 TB** (`sim/smoke/`, flash 목 포함): 27체크 전부 PASS — ID/리셋값, INC·DEC·signed PHASE_POS, ps_busy·meas_busy 중 CTRL 무시(R2·R4), 다중 비트 무시, START→BUSY→DONE→sticky→다음 START 클리어(R5), ERR·LOG 창(읽기 부작용 없음), CFG_ERR 거부 시 busy/done 미발행(R11), TIMEOUT sticky와 유효 START 클리어(R10)
  2. **xelab + unisims_ver**: 실물 MMCME2_ADV 모델 기준 elaboration 통과 (포트·파라미터 정합)
  3. **Vivado OOC 합성** (xc7z020clg400-1): synth_design 통과 확인
- AXI 슬레이브는 단일 미결(outstanding 1)·항상 OKAY 응답 — 폴링 트래픽(8.4ms당 ~200회)에 충분, LOG_DATA는 별도 트랜잭션이라 "주소 다음 사이클 유효" 계약을 자동 충족

## 미결 의사결정

- **계약 수정 제안 4건 — contract.md에 "※7/8 추인 대상" 표기로 반영 완료 (7/7 밤), 승인만 남음**:
  1. §2 신호 2개: `meas_timeout`(done과 동시 1펄스, R10)·`cfg_err`(거부 1펄스, R11) — STATUS.TIMEOUT/CFG_ERR은 core 레지스터인데 판정 주체는 flash라 전달 경로가 §2에 없었음. 부결 시 core가 R11을 자체 검사하는 대안으로 재작업 필요(R10은 어차피 신호 필요)
  2. §3 CTRL "명령 비트는 한 번에 하나" + STATUS bit6 `CMD_ERR`(sticky, 리셋으로만 클리어) — 다중 비트 쓰기는 호스트 버그이므로 시끄러운 거부(R11 철학)
  3. R11 "유효 START" 조작적 정의 = `meas_busy` 상승 에지 — 거부된 START가 이전 TIMEOUT 증거를 못 지움
  4. §6 무효 런 ⑥ 스윕 종료 시 CMD_ERR=1 — 호스트 스윕 코드가 종료 시 1회 확인
- sticky 클리어 시점 중 DONE=START 수락 시(R5 문언 그대로)는 계약 수정 아님 — 추인 때 통보만
- MMCM 파라미터 중 O=45는 브링업 보수 선택 — 안정화 후 O=25/15 상향 검토(결정 16)는 core_mmcm.v 파라미터 2줄 수정으로 끝남
- **flash 구현 노트 (자체 재리뷰에서 도출)**: flash는 수락한 start의 **다음 사이클에 meas_busy를 올려야** 한다 — core의 명령 게이팅이 meas_busy를 참조하므로, busy 상승이 2사이클 이상 늦으면 그 틈에 두 번째 START가 수락될 수 있는 창이 생긴다(R11 검증은 cfg 래치값의 조합 논리라 1사이클 판정 가능, R11의 "busy 발행 없음"과도 양립). 호스트가 R4대로 STATUS 확인 후 명령하면 실제로는 안 밟는 창이지만, 하드웨어 성질로 닫아두는 게 계약 철학과 일치
