# 0. 위상 아키텍처 — 샘플 클럭 시프트 vs SCLK 시프트

- 상태: **확정 — A안(샘플 클럭 시프트) 채택** (2026-07-07, 한영웅). 수정은 3인 합의로만.
- 작성: 한영웅, 2026-07-07

## 후보

| 안 | 무엇을 시프트하나 | 판정 |
| --- | --- | --- |
| **A. 샘플 클럭 시프트** | SPI 마스터(SCLK·CS·MOSI)는 고정 클럭 `clk_core`. MISO **캡처 플롭만** MMCM 동적 위상 시프트 출력 `clk_sample`로 구동 | **채택** |
| B1. SCLK 출력만 시프트 | SCLK 핀만 시프트, 마스터 로직·MOSI·CS는 고정 | **기각** — 시프트가 커지면 MOSI·CS ↔ SCLK 상대 타이밍이 함께 틀어져 명령 자체가 깨질 수 있음. 에러가 읽기 마진인지 명령 오염인지 구분 불가 (교란 변수) |
| B2. SPI 마스터 전체 시프트 | SCLK·CS·MOSI를 전부 시프트 도메인에서 생성, MISO 캡처는 고정 도메인 | 물리적으로 A와 등가 측정. 단 시프트 클럭이 SPI 마스터 전체 + 출력 핀을 구동해 CDC 표면·타이밍 제약이 A보다 큼 |

## A안 권고 근거

1. **DUT 자극 완전 불변** — 칩이 보는 SCLK·CS·MOSI가 모든 위상 스텝에서 동일. 곡선의 변화를 캡처 타이밍 하나에만 귀속시킬 수 있다 (계측의 통제 변인).
2. **시프트 도메인 최소화** — 시프트 클럭이 구동하는 것은 캡처 플롭 + 비교기·카운터 경로뿐. SPI 마스터는 단일 고정 도메인(flash 서브시스템 설계 단순).
3. **루프백 G0와 실칩 경로가 동일 구조** — 출력 핀(고정 도메인 패턴) → 입력 핀(시프트 캡처). 엔진 재사용.
4. 프로젝트 서사와 일치 — "칩이 아니라 측정기가 경계까지 다가간다" (`project_context.md` §3.2).

## 설계 메모 (계약 아님 — G0 RTL 유의사항)

- `clk_core`/`clk_sample`은 동일 주파수·가변 위상 (mesochronous). 위상 이동은 **버스트 사이(CS high)에서만** 수행.
- 캡처 스트림의 비트 프레이밍은 버스트마다 재정렬 (도메인 간 ±1 사이클 지연 변동 대비).
- 측정 결과는 done 시점에 core 도메인으로 핸드셰이크 전달 — 계약 1 참조.

## 파급

이 결정이 계약 1(핸드셰이크 클럭 도메인), 2(레지스터 의미), 4(`phase_step` 정의)의 전제다. 변경 시 셋 다 재검토.

# 1. core ↔ flash 핸드셰이크 — 결정 항목

- 상태: **확정** — 1-1 (a)·1-2 (a)·1-3 (a) 전항 확정, 3-4(A) 반영으로 `err_sq_sum`·`cnt_ovf` → 버퍼 읽기 포트 대체 (2026-07-07). 수정은 3인 합의로만.

| --------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------- |
| (a)**flash 측이 캡처~집계+CDC 전부 소유.** 계약 경계는 clk_core 단일 도메인 | 계약이 가장 단순(전 신호 한 도메인). 캡처·비교·카운터가 한 소유자라 프레이밍 재정렬을 내부에서 해결 | 장세은이 mesochronous CDC를 설계해야 함 (부담 이동)                                                      |

**→ 결정: (a) 확정** (2026-07-07). CDC 템플릿(2단 동기화 + done 핸드셰이크)은 한영웅 제공.

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| (a) **호스트(PS) 주도** — 스텝당 {PHASE_INC → 폴 → MEAS_START → 폴 → 카운터 리드} | RTL 최소 (자율 FSM·결과 버퍼 불요) → 7/10 크리티컬 패스 단축. 스텝 단위 관찰 가능해 디버깅 쉬움. 속도 손실 무시 가능 — 스텝당 측정 8.4ms(25MHz·N=100)가 지배적, AXI 폴링은 µs 단위 | 스윕 시퀀싱 책임이 PS 소프트웨어로 이동 (펌웨어 소유자와 조율 필요). 스텝 간 간격이 소프트웨어 지터에 노출 (측정 자체는 HW라 데이터 오염은 없음) |

3-4 (A) 반영 (2026-07-07): `ERR_SQ_SUM`·`STATUS.CNT_OVF` 삭제 — 전 카운터가 구조적 오버플로 불가라 포화·OVF 개념 자체가 소멸. 제곱합은 호스트가 버퍼에서 파생 계산해 CSV에 기록.

부속 규약: busy(MEAS/PS) 중의 CTRL 명령은 무시 — 호스트는 STATUS 확인 후 명령. 결과 레지스터는 DONE~다음 START 구간 안정. 미정의 비트 RAZ/WI.

# 3. 에러 카운터 폭 + B_BITS — 결정 항목

- 상태: **확정** — 3-1 32비트 통일 · 3-2 포화+OVF(원칙) · 3-3 B_BITS=2,048 · 3-4 (A) 읽기별 버퍼, 전항 확정 (2026-07-07). 수정은 3인 합의로만.
- 기준 수치: 위상 스텝당 최대 비트 에러 = N×B = 100 × 2,048 = **204,800 → 최소 18비트**

## 결정 3-1. `err_bits` 카운터 폭

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| 18비트 (최소) | 자원 최소 (미미한 차이) | 여유 0 — N이나 B를 조금만 올려도 재계약. 실측 σⱼ로 N 재확정(G0 체크 3항) 예정이라 위험 |
| **24비트** | N×B 대비 ×81.9 여유 (B=2,048이면 N ≤ 8,191까지 무재계약). 상한이 계약에 명시적으로 드러남 | 32비트 대비 특별한 자원 이득은 없음 |
| 32비트 (레지스터 전폭) | 폭 고민 자체를 제거 | "얼마까지 유효한가"가 계약에서 사라짐 — 상한 인식 없이 N·B를 키우다 스텝당 시간 폭증을 늦게 발견하는 류의 실수 여지 |

**추천: 24비트.** 자원 차이는 셋 다 무의미하고, 실질 쟁점은 "상한을 계약에 남기느냐"다. 24비트는 충분한 여유와 명시적 상한을 동시에 준다.

**→ 결정: 32비트 통일** (2026-07-07, 24비트 추천 철회). 근거: 설정 필드가 16비트(N, B ≤ 65,535)이므로 err_bits 최대 = 65,535² < 2³² — **구조적으로 오버플로 불가**, 포화 로직도 불필요해져 RTL이 오히려 단순. err_reads도 동일. "상한 의식"은 설정 필드 폭과 스윕 시간이 이미 담당. 포화+OVF(3-2)는 err_sq_sum 전용으로 남는다(극단 설정에서 N×B² > 2³² 가능).

## 결정 3-2. 오버플로 거동

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| **포화(saturate) + STATUS.CNT_OVF sticky** | 오염이 눈에 보이고(값이 천장에 붙음) 플래그로 해당 스텝 무효 판정 가능 | 포화 로직 약간 추가 (비교기 1개) |
| 랩어라운드 (자연 오버플로) | RTL 추가 없음 | **욕조 곡선 어깨를 조용히 왜곡** — 에러 많은 스텝이 에러 적은 값으로 둔갑. PASS/FAIL이 아니라 값 오염이라 검증에서도 안 잡힘 (MC 스크립트 `!!` 플래그가 경고한 바로 그 실패 모드) |
| 오버플로 시 측정 즉시 중단(abort) | 오염 데이터 원천 차단 | 중단 상태·재시작 경로가 FSM에 추가. 곡선 극단(BER~0.5)에서는 포화가 정상 상황이라 중단이 오히려 스윕을 막음 |

**추천: 포화 + 플래그.** 랩만은 반드시 배제. 극단 위상에서는 포화가 일어나도 되므로(그 스텝 BER은 어차피 0.5 포화 해석) abort는 부적합 — 플래그를 보고 호스트가 판단한다.

**→ 결정: 포화 + OVF 확정** (2026-07-07). ※ 3-1이 32비트로 확정되면 err_bits·err_reads는 구조적으로 오버플로 불가가 되어, 이 규약은 실질적으로 err_sq_sum에만 적용된다. (후기: 3-4가 (A)로 확정되며 sq_sum도 하드웨어에서 사라져 현재 적용 대상 없음 — 오버플로 가능한 카운터가 새로 생기면 이 규약을 적용하는 **원칙**으로 유지.)

## 결정 3-3. B_BITS (읽기 1회 버스트 길이) 확정

지금까지 추론값(`!!` 플래그) — 이 계약에서 확정한다. 제약: 최저 병행 θ=10⁻⁴의 유효 조건 **N×B×θ ≥ ~20**.

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| 512비트 (64B) | 스텝당 시간 1/4 | 조건 충족에 N=400 필요 → 총 비트 수 동일한데 읽기 오버헤드(40비트/회) 비중만 4배 — 순손해 |
| **2,048비트 (256B, 1페이지)** | MC의 N=100 권고와 정합 (N×B×θ=20.5). 페이지 단위라 주소 관리 자연스러움. 스텝당 8.4ms·스윕 ~23초(25MHz)로 실용적 | — |
| 4,096비트 (2페이지) | 읽기당 오버헤드 비중 절반, 같은 총 비트에 N=50 | MC 재실행 필요 (N 권고 연쇄 변경). 통계 이득은 총 비트 수가 같으면 없음 |

**추천: 2,048비트.** 이미 모든 분석이 이 값 위에 서 있고, 대안들의 이득이 미미하거나 음수다. 확정 시 `monte_carlo_sweep_params.py`의 `!!` 플래그 해소.

**→ 결정: 2,048비트 확정** (2026-07-07). 결정 근거에 페이지 프로그램 단위(256B)와의 1:1 대응 포함. `!!` 플래그 해소 — 스크립트 docstring 동기화 완료.

## 결정 3-4. `err_sq_sum` (읽기별 에러 수 제곱합) 채택 여부 — **박지민 합의 필요**

용도: 에러 버스트성 점검 = θ 확정 체크리스트 4항 (비트 독립 가정 검증, `bathtub_analysis.py`가 소비).

| 후보 | 장점 | 단점 |
| --- | --- | --- |
| **채택** — error logger에 제곱 누적기 추가 | 체크 4를 계획대로 정량 판정 (과분산 F 통계) | logger에 곱셈기(11×11비트)+32비트 누적기 추가 — 박지민 작업량 증가 |
| 미채택 — `reads_with_error`로 근사 | RTL 추가 최소 (비교기 1개) | 버스트성 검정력이 낮음 — "에러 있는 읽기 수 vs 이항 기대치" 비교는 약한 버스트만 있으면 못 잡음 |
| 보류 — 신호·레지스터만 예약, 구현은 G0 이후 | 7/10 크리티컬 패스에서 제외하면서 계약 재협상 없음 | G0 데이터로 체크 4를 못 돌림 → θ 확정이 그만큼 지연 |

**추천: 채택. 단 박지민 부담이 크면 보류(예약)로.** 곱셈기라 해도 DSP 슬라이스 1개 수준이고, θ 확정 체크리스트가 G0 직후 예정이라 G0 데이터에 제곱합이 없으면 체크 4만 뒤로 밀린다.

**추가 후보 (A) — 읽기별 에러 수 버퍼** (2026-07-07 논의에서 추가): 제곱합 누적 대신 읽기별 에러 수 e_i (≤ B=2,048)를 그대로 작은 RAM(128×16b)에 기록하고, 스텝 종료 후 호스트가 레지스터 창으로 회수. 제곱합은 호스트가 파생 계산(CSV·분석기 불변). 정보 포함 관계가 일방향(A→제곱합 가능, 역 불가)이라 **결과 품질은 A가 우위** — 원시값 보존, 사후 재분석·이상 스텝 원인 추적 가능, 각 항목이 2,048 이하라 포화 규약 자체가 불필요. 비용: 레지스터 맵 읽기 창 추가(계약 2 개정) + 호스트 회수 코드(주로 한영웅 몫), 스텝당 시간 +1% 미만. 박지민 몫 RTL은 곱셈기→RAM 기록으로 오히려 단순.

**→ 결정: (A) 읽기별 에러 수 버퍼 확정** (2026-07-07). 기준 = 결과의 품질(원시값 보존 — 정보 포함 관계가 A→제곱합 일방향, 사후 재분석·이상 추적 가능, 포화 개념 소멸) > 구현 난이도. 제곱합은 호스트 파생으로 이동 — CSV 열·분석기 불변. 박지민 인계 사양 = "읽기 종료마다 e_i를 RAM에 기록"(곱셈기 불요). 계약 1 포트 교체·계약 2 창 추가 반영 완료.

## 전제: O는 정수, 따라서 f는 VCO(1,125MHz)의 약수로 한정

VCO는 D=1·정수 M 제약(지터 최소화, `2.phase_sweep_and_mmcm.md` 참조)으로 이미 1,125MHz 확정. MMCM의 O는 정수 손잡이라 출력 주파수 f = 1,125 ÷ O는 **1,125의 약수만 가능**하다 (1,125 = 3²×5³ → 1, 3, 5, 9, 15, 25, 45, 75, 125, 225, 375, 1125 MHz). 애초 몬테카를로 비교에 쓴 25/50/100MHz 중 **50·100은 약수가 아니라 애초에 불가능한 값**이었다 — 이번 결정으로 그 사실이 확정에 반영됨.

## 후보 (실용 SPI 클럭 대역 내 약수만)

스윕 시간은 스텝 수(56×O)와 스텝당 읽기 시간이 둘 다 f에 반비례해 **f² 꼴로 줄어든다** (25MHz 22.8초 → 100MHz 1.5초 실측 비율 확인, `monte_carlo_and_artifact_policy.md` 참조). 이 관계로 역산:

| O | f | 스윕 시간(N=100, 추정) | 판정 |
| --- | --- | --- | --- |
| 75 | 15MHz | ~63초 | 기각 — 불필요하게 느림 |
| **45** | **25MHz** | **~23초** | **채택 (v0, 브링업)** |

## 권고 근거

G0~G4(중간보고서 마일스톤)는 "실칩 저클럭 브링업"(G2)을 포함한 초기 검증 단계다. 이 단계의 목적은 스윕 속도가 아니라 **MMCM 위상 제어·핸드셰이크가 의도대로 도는지 확인**이므로, 후보 중 가장 느리고 널널한 25MHz로 시작해 타이밍 여유를 확보한다. 스윕 시간(약 23초)은 이 단계에서 감내 가능한 비용.

## 향후 재검토 (미결)

프로젝트가 안정화되고(G4 이후, 반복성 확보 후) 스윕 속도가 병목이 되면 O=25(45MHz) 또는 O=15(75MHz)로 상향을 검토한다. 상향 시 재작업 범위:
- MMCM O 설정 변경 (M=9·D=1·VCO=1,125MHz는 불변 — Δφ=15.87ps 그대로)
- `monte_carlo_sweep_params.py` 재실행 (스윕 시간 모델의 f_sclk 반영)
- 계약 수치(`f_sclk_hz`, 스윕 소요시간 추정치) 갱신, 3인 합의 필요 (계약 변경 원칙)

Δφ(15.87ps)·2,520스텝 자체는 O 값과 무관하게 O=45 기준이므로, O를 바꾸면 UI당 스텝 수(56×O)도 함께 바뀐다는 점에 유의 — 예: O=25면 56×25=1,400스텝.
