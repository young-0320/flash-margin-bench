# 4. AXI-Lite — 프로세서가 우리 회로를 조작하는 리모컨의 규격

- 관련 결정: 계약 2(레지스터 맵 v0), 2-1(호스트 주도 스윕)

## 버스: 칩 안의 표준 택배 규약

Zynq 칩 안에는 프로세서(PS, ARM 코어)와 우리가 만든 회로(PL, FPGA 로직)가 같이 산다. 이 둘이 대화하려면 "주소를 말하면 값을 주고받는다"는 공통 규약이 필요한데, 그 표준이 **AXI**다. 소프트웨어 입장에서 우리 회로는 그냥 **메모리의 한 구역**처럼 보인다 — 특정 주소를 읽으면 우리 회로의 상태가 나오고, 특정 주소에 쓰면 우리 회로가 동작한다. C 코드로는 포인터 접근 한 줄이다.

AXI에는 용도별 변종이 셋 있다:

| 변종 | 비유 | 용도 |
| --- | --- | --- |
| **AXI-Lite** | 리모컨 + 계기판 | 소량의 제어·상태를 주소로 읽고 쓰기 (한 번에 32비트 하나씩) |
| AXI4 (Full) | 화물 트럭 | 주소 기반 대량 전송 (버스트 — 한 번의 주소 지정으로 연속 데이터) |
| AXI-Stream | 컨베이어 벨트 | 주소 없음. 한 방향으로 데이터가 연속으로 흐름 (DMA와 함께 씀) |

우리 트래픽은 "8.4ms마다 버튼 몇 번 + 계기판 몇 칸 읽기"라서 리모컨(Lite)이 정확히 맞는 도구다. 컨베이어 벨트(Stream)는 초당 수십 MB가 흐를 때 존재 이유가 생기고, 그때도 리모컨은 항상 따로 있어야 한다 — "시작해", "바쁘냐" 같은 양방향 대화는 벨트로 표현할 수 없기 때문이다.

## 레지스터 맵: 리모컨의 버튼 배치도

AXI-Lite로 노출하는 주소들과 각 주소의 의미를 정한 표가 **레지스터 맵**이다. 우리 계약의 맵을 유형별로 읽으면:

- **버튼 (쓰면 동작)**: `CTRL` — 측정 시작, 위상 한 칸 이동. 쓰는 순간 1사이클 펄스로 바뀌고 스스로 0으로 돌아간다(self-clearing). 버튼은 눌렀다 떼는 것이지 눌러둔 채 두는 게 아니라는 뜻.
- **계기판 (읽기 전용)**: `STATUS`(바쁨/완료/이상), `PHASE_POS`(현재 위상 위치), `ERR_BITS` 등 측정 결과.
- **다이얼 (설정)**: `N_READS`, `BURST_BITS` — 읽고 쓸 수 있고, 측정 시작 순간의 값이 스냅샷으로 잡힌다.
- **열람창**: `LOG_ADDR`에 번호를 쓰고 `LOG_DATA`를 읽으면 그 번호 읽기의 에러 수가 나온다 — 큰 데이터(버퍼)를 32비트 창으로 한 칸씩 들여다보는 표준 수법.

## 설계 감각 몇 가지 (오늘 결정들의 배경 원리)

**sticky 플래그**: 완료·이상 표시등은 한 번 켜지면 유지된다. 소프트웨어는 폴링(주기적으로 들여다보기)으로 상태를 확인하는데, 순간만 켜지는 신호는 폴링 사이에 놓칠 수 있기 때문이다. 우리는 끄는 방식도 "다음 시작이 자동으로"로 정해서, 소프트웨어가 지우기를 빼먹어 이전 완료를 새 완료로 오독하는 고전 버그를 원천 차단했다.

**ID 레지스터**: 주소 0에 고정된 매직 넘버를 둔다. 브링업 첫날 "읽어서 0x4D420100이 나오는가"만 확인하면 주소 연결·버스 배선 전체가 한 번에 검증된다 — 가장 싼 스모크 테스트.

**읽기의 부작용 금지**: 읽기만 했는데 상태가 바뀌는 레지스터(예: 읽으면 자동으로 다음 칸으로 넘어가는 창)는 디버거로 들여다보는 것만으로 상태가 망가진다. 그래서 우리 열람창은 주소를 명시적으로 쓰는 방식이다 — 몇 번을 다시 읽어도 같은 값.

**접근 속도의 함정**: PS에 리눅스를 올리면 이 주소들에 접근하는 방법이 여러 가지인데 속도가 수백 배 차이 난다(포인터 직접 접근 ~0.1µs vs 파일시스템 경유 수십 µs 이상 — 우리는 스텝당 200회쯤 접근). 우리 결정(2026-07-07)은 **bare-metal C** — OS 없이 PS에서 바로 실행하므로 레지스터가 처음부터 그냥 메모리 주소라 함정 자체가 없고, 부팅·드라이버 준비 비용도 0이다. 리눅스가 필요해지는 시점(파일시스템·네트워크)은 스프린트 밖이다.
